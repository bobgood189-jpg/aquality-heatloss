# PROGRESS — починка бэкенда (Supabase)

Статус: **код реализован и проверен в браузере на всём, что проверяемо без боевых
ключей**. Боевой сетевой round-trip к Supabase требует, чтобы владелец вставил
`SUPABASE_URL` + `anon key` и выполнил `supabase/schema.sql` (см. `README-SUPABASE.md`).

Архитектура: единый слой данных через Supabase с graceful-fallback на localStorage,
когда ключи не заданы (математика работает всегда). Подробности — `BACKEND_AUDIT.md`.
Доказательства проверок — `test-screens/EVIDENCE.md`.

## ❗ Что должен сделать владелец (приёмка боевого режима)
1. Создать проект Supabase, выполнить `supabase/schema.sql` (SQL Editor).
2. Вставить `SUPABASE_URL` + `anon key` в `window.AQ_CONFIG` (верх `index.html`).
3. Зарегистрироваться → выполнить `update profiles set role='owner' where email='...'`.
4. (Опц.) Отключить «Confirm email» для входа без письма.
5. Приёмочный тест в браузере (как в DoD):
   - Регистрация → вход → F5 → всё ещё в аккаунте → выход → профиль недоступен.
   - Профиль: изменить → F5 → данные на месте; в инкогнито под другим аккаунтом —
     видны только свои данные.
   - Сохранить расчёт → «Мои расчёты» → F5 → на месте → открывается с теми же данными.
   - Отправить заявку → строка появляется в Supabase → `select * from leads;` →
     WhatsApp открылся. Под не-админом `select` по `leads` ничего не отдаёт.

## Follow-up: меню владельца «полноценно рабочим» (офлайн-режим)
Владелец (`__owner__`) — синтетический аккаунт, его не было в `aq_users_v3`, поэтому
профиль был только для чтения, а расчёты не сохранялись. Добавлено отдельное хранилище
`aq_owner_v1` (`ownerStore`/`ownerStoreSave`). Теперь для каждого пункта дропдауна:
- **Мой профиль** — редактируется и сохраняется (имя/телефон/телефон2/компания/город),
  переживает перезагрузку.
- **Мои расчёты** — «Сохранить в кабинет» + автосохранение на шаге 6 → список →
  открытие → удаление; всё переживает перезагрузку.
- **Панель администратора** — дашборд/пользователи/заявки рендерятся (async-safe).
- **Выйти** — работает.

Проверено в браузере (как владелец, офлайн): профиль и расчёты сохраняются и видны
после F5; сессия владельца восстанавливается; консоль чистая; математика = 6.3601 кВт.
В Supabase-режиме владелец = `profiles.role='owner'`, всё это и так работало через БД.

## Известные ограничения
- Воронка-аналитика (`aq_events_v1`) и анонимная история (последние 10) остаются в
  localStorage — это вспомогательная статистика, не критичные данные. Можно вынести
  в Supabase отдельной задачей при необходимости.
- Смена роли пользователя (admin) в Supabase-режиме делается через SQL (безопасно),
  не из UI — кнопка «назначить админом» в UI активна только в офлайн-режиме.
- Реальный сетевой round-trip к Supabase в этой сессии не выполнялся (нет боевых
  ключей) — см. раздел «Что должен сделать владелец».
