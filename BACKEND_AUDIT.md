# BACKEND_AUDIT.md — состояние слоя данных ДО починки

Аудит выполнен по коду `index.html` (single-file, ~4632 строки) перед интеграцией Supabase.

## 1. Авторизация — псевдо-авторизация на localStorage

- **Где:** `index.html`, блок `AUTH MODULE` (строки ~650–882).
- **Хранилище пользователей:** `localStorage['aq_users_v3']` (`AUTH_USERS_KEY`). Массив объектов `{id, login, password_hash, phone, phone2, email, city, clientType, role, createdAt, projects[]}`.
- **Сессия:** `localStorage['aq_session_v3']` (`AUTH_SESSION_KEY`) — хранит `id` пользователя или строку `'__owner__'`.
- **Пароли:** функция `_hashPass(p) = btoa(encodeURIComponent(p)+'|aq2')` — это **НЕ криптография**, тривиально обратимо. Раньше пароли вообще хранились в открытом виде (есть legacy-ветка миграции plaintext→hash).
- **Владелец:** логин `Hasanboy.admin`, пароль захардкожен `atob('UC5tYXhtdWQ3NA==')` = `P.maxmud74` прямо в JS-бандле. Любой может открыть исходник и войти владельцем.
- **Вывод:** «аккаунты» существуют только в браузере одного устройства. На другом устройстве/в инкогнито пользователя нет. Данные легко стираются, пароли не защищены. Это не бэкенд.

## 2. Профиль

- **Где:** `openProfile()` (строки ~919–955). Профиль — это просто поля того же объекта в `aq_users_v3`.
- Экран профиля **только для чтения** (логин, тип, дата регистрации, число проектов) + кнопки «Мои расчёты» и «Выйти». Редактирования профиля нет.
- **Теряется:** при очистке localStorage, на другом устройстве, в другом браузере — профиля нет.

## 3. Сохранённые расчёты / история

- **История:** `localStorage['aq_history_v1']` (`HIST_KEY`), последние 10, анонимная (не привязана к пользователю), функции `loadHistory/pushHistory/loadFromHistory/clearHistory` (строки ~4001–4021).
- **Проекты пользователя:** `authSaveProject(label)` пишет в `user.projects` внутри `aq_users_v3` (строки ~733–743). Показываются в «Мои расчёты» (`openMyCalcs`, ~957–992).
- **«Сохранить расчёт (ссылка)»:** `saveLink()` (~3983) — кодирует состояние в URL-хэш `#r=...`, кладёт в буфер обмена. Не серверное хранилище.
- **Вывод:** ничего не уходит на сервер. Перезагрузка на другом устройстве — расчёты исчезают.

## 4. Лиды (заявки WhatsApp/Telegram)

- **Где:** `sendWhatsApp()` (строки ~4196–4223) и `saveLead()` (~1224–1229).
- Лид пишется в `localStorage['aq_leads_v1']` (`LEADS_KEY`) **и** открывается `wa.me/<номер>` с готовым текстом.
- **Проблема для бизнеса:** заявки видны только в том браузере, где их отправили. Владелец на своём устройстве их НЕ увидит (это другой localStorage). Реально заявка «доходит» только если клиент дожмёт кнопку «Отправить» в WhatsApp. Серверного хранилища заявок нет — заявки теряются.
- Telegram-канал в коде не реализован (только WhatsApp + `tel:`).

## 5. Supabase / зависимости

- `@supabase/supabase-js` **НЕ подключён**. Таблиц, ключей, RLS — нет.
- CDN в `<head>`: Tailwind Play CDN, GSAP+ScrollTrigger, Lenis. Бандлера/`package.json`/`node_modules` нет — чистая статика.
- Воспроизведение багов:
  - «вход → F5 → разлогинило»: НЕ воспроизводится буквально (сессия в localStorage переживает refresh), НО на другом устройстве/в инкогнито аккаунта нет вовсе — для пользователя это выглядит как «слетела сессия / пропал аккаунт».
  - «сохранил расчёт → пропал»: воспроизводится при смене устройства/браузера или очистке кэша — данные только локальны.
  - «заявка не дошла»: воспроизводится — владелец не видит заявок, отправленных с чужих устройств.

## Решение по архитектуре

Текущее состояние = localStorage-псевдобэкенд. **Мигрируем на Supabase** (Auth + Postgres + RLS) как единый слой данных, сохраняя single-file подход (Supabase JS через CDN UMD).

Ключевые инженерные решения:

1. **Конфиг** — `window.AQ_CONFIG` (URL + anon-ключ) в начале `index.html` с плейсхолдерами (дублируется в `config.js` как шаблон). Только `anon`-ключ на клиенте; `service_role` — никогда.
2. **Синхронный кэш сессии** (`_AQ.user`) — Supabase Auth асинхронен, а в коде десятки синхронных вызовов `authCurrentUser()`. Чтобы не переписывать все call-site'ы на async, при загрузке гидрируем кэш через `getSession()` + `onAuthStateChange`, а `authCurrentUser()` остаётся синхронным и читает кэш. Это чинит «слетает сессия» — Supabase сам хранит сессию и поднимает её на старте.
3. **Роутер «Supabase ↔ legacy»** — когда `SUPABASE_*` заданы, активен ТОЛЬКО путь Supabase (одна точка правды). Когда не заданы — graceful degradation: математика работает полностью, а аккаунт/лиды откатываются на старый localStorage-путь, чтобы сайт не «умер» до вставки ключей. Это исключает два одновременных источника правды.
4. **Миграция (E)** — при первом входе в Supabase предлагаем перенести локальную историю расчётов в `calculations`, затем ставим флаг `aq_migrated_v1`.

Что НЕ трогаем: математику КМК (`computeObject`/`computeRoom`), 2D/3D-редактор, GSAP/ScrollTrigger/Lenis, премиум-вид.
